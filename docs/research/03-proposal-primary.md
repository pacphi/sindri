# Primary Recommendation — "Sindri BOM"

**One-line pitch.** A Sindri v4 project declares a `sindri.yaml` BOM manifest of
atomic, OCI-addressable components; each component entry is a
`backend:name@version` triple pointing into a curated per-backend registry;
collections are meta-components; the compatibility matrix is deleted; the resolved
manifest _is_ the SBOM.

This synthesizes mise (manifest syntax), devcontainer features (component shape +
OCI distribution), aqua (registry discipline), apt (meta-package collections), and
apko (SBOM as output).

---

## 1. User-facing manifest: `sindri.yaml`

```yaml
# sindri.yaml — authored by the user, checked into the project
apiVersion: sindri.dev/v4
kind: BillOfMaterials

# Which curated registries (catalogs) to resolve against.
# Each is content-addressable (tag OR digest).
registries:
  - oci://ghcr.io/sindri-dev/registry-core:2026.04
  - oci://ghcr.io/sindri-dev/registry-ai:2026.04
  - oci://ghcr.io/acme/registry-internal:v7 # org-private

# What to install. Each entry resolves to exactly one component version.
components:
  # Runtime — backend explicit, version pinned.
  mise:python: "3.14.0"
  mise:nodejs: "22.11.0"
  mise:golang: "1.24.1"

  # Atomic AI CLIs (replacing v3 ai-toolkit bundle).
  npm:codex@openai: "2.3.1"
  npm:gemini-cli@google: "0.8.4"
  binary:fabric: "1.4.17"

  # Cloud CLIs pick-your-own (replacing v3 cloud-tools bundle).
  binary:aws-cli: "2.17.21"
  mise:gcloud: "499.0.0"

  # System packages still supported.
  apt:docker-ce: "27.3.1-1~ubuntu.24.04~noble"

  # A collection (meta-component) — pulls its own dependsOn.
  collection:anthropic-dev: "2026.04"

  # Component with options (typed per-component schema).
  mise:ruby:
    version: "3.3.6"
    options:
      yjit: true

# Post-install project wiring (unchanged from v3 project-init semantics).
project:
  init:
    - component: npm:agentic-qe
      requiresAuth: anthropic
```

**Design properties:**

- Map keys are `backend:name` — same "backend:tool" idiom users already know from mise.
- Values are either a pinned version string or `{ version, options }`.
- Registries are OCI artifacts, tag- or digest-addressable. Corporate forks are trivial.
- No global "channel" / "CLI version" field — components advance independently.

## 2. Lockfile: `sindri.lock`

Generated by `sindri resolve`. Records, for each component:

- The backend it resolved against.
- The exact upstream version.
- The OCI digest of the component definition in the registry.
- The checksum (sha256) of every download artifact referenced.
- The `dependsOn` closure, fully expanded.

Renovate-updatable via inline hints on the manifest and/or a lockfile parser plugin.

## 3. Component shape (replaces today's extension.yaml)

```yaml
# In the registry OCI artifact: components/nodejs/component.yaml
apiVersion: sindri.dev/v4
kind: Component

metadata:
  name: nodejs
  backend: mise # primary backend this component is published to
  versions: # supported upstream versions
    - "20.x"
    - "22.x"
  category: languages

# What the user sees when selecting.
options:
  corepack:
    type: bool
    default: true

# Dependency DAG.
dependsOn:
  - mise-config

# How to install. Exactly one of:
install:
  mise:
    tools:
      - "node@{{ version }}"
    reshim: true
  # OR apt / binary / npm / script — one backend per component.
  # Hybrid is GONE. If you need two steps, compose two components.

# Lifecycle phases (unchanged in spirit from v3):
configure: { ... }
validate: { ... }
remove: { ... }

# Capabilities (unchanged from v3, reduces churn on what works):
capabilities:
  hooks: { ... }
  project-init: { ... }
  collision-handling: { ... }
  mcp: { ... }
```

**Key changes vs v3 extension.yaml:**

- `install.method` enum is replaced by **one install block whose shape is dictated by the backend**.
- `InstallMethod::Hybrid` is deleted. A v3 hybrid extension becomes two atomic components with a `dependsOn` edge (e.g., `apt:docker-ce` + `script:docker-post-install`).
- `bom.tools[]` is deleted from the component definition. The resolver emits the SBOM; authors don't re-declare what they just declared.
- `metadata.version` (of the _extension package_) is replaced by the component's OCI digest.

## 4. Registries: curated catalogs, one per backend

Published as OCI artifacts. Structure:

```
ghcr.io/sindri-dev/registry-core:2026.04
├── index.yaml              # component → versions → digest map
├── components/
│   ├── nodejs/component.yaml
│   ├── python/component.yaml
│   └── …
└── checksums/              # per-artifact sha256
```

Invariants enforced at registry-publish time (CI):

- **Forced pinning:** every version in `index.yaml` must be exact, not a range.
- **Checksums required** for any downloaded artifact (script, binary, tarball).
- **`# renovate: depName=…` hints** in `component.yaml` source fields.
- **Reproducibility:** re-resolving the same tag produces identical results.

Sindri ships a "core" registry. Orgs publish their own. Users list the registries they
trust in `sindri.yaml` — same model as aqua's `registries:`.

## 5. Collections — meta-components, not a separate type

```yaml
# components/anthropic-dev/component.yaml
apiVersion: sindri.dev/v4
kind: Component
metadata:
  name: anthropic-dev
  type: meta # no install block; only dependsOn
dependsOn:
  - mise:nodejs
  - mise:python
  - npm:codex@openai
  - npm:claude-code
  - binary:fabric
```

This unifies profiles and bundles into one abstraction. v3's `profiles.yaml` becomes
unnecessary. Users author their own meta-components in their own registries for
team-wide "opinionated stacks."

A **dynamic-collection** variant (Renovate-style predicate: `packageRules` + `groupName`)
is proposed but optional — see alternatives.

## 6. Install pipeline

```
sindri.yaml
   │
   ▼  sindri resolve
sindri.lock  ◀──── registries (OCI, cached, verified by digest)
   │
   ▼  sindri install
per-backend installer (mise | apt | binary | npm | script)
   │
   ▼  capabilities: configure → validate → hooks → project-init → collision-handling
   │
   ▼  sindri bom  (SPDX / CycloneDX — generated from sindri.lock, not re-declared)
```

The dispatch table shrinks: one handler per backend, no `Hybrid`, no per-distro switch
inside a single component (distro-variance is handled by the backend or by picking a
different component in the manifest).

## 7. What gets deleted from v3

- `v3/compatibility-matrix.yaml` — gone. Components are pinned independently.
- `v3/docs/CLI_EXTENSION_COMPATIBILITY_GUIDE.md` as a maintained table — gone. The CLI and registries version independently.
- `v3/profiles.yaml` — replaced by meta-components.
- `InstallMethod::Hybrid` — replaced by component composition.
- `bom.tools[]` inside components — replaced by resolver-generated SBOM.
- `sindri-projects` template-to-extension coupling — projects become optional meta-components plus a scaffold template.
- "Bundle extensions" `ai-toolkit`, `cloud-tools`, `infra-tools` — decomposed into atomic components; optional meta-components preserve the "install the whole set" UX.

## 8. What stays largely unchanged

- Capability system (hooks, project-init, collision-handling, mcp, project-context) — same schemas, just hosted inside the new component shape.
- StatusLedger event model for install/upgrade/remove/validate events.
- Scripted-install escape hatch — still a backend (`script:`), just no longer combined with other backends inside one component.
- Extension authoring UX at the YAML level is highly familiar — the rename is mostly cosmetic.

## 9. Answering the user's stated goals

| Goal                                                                     | How this delivers                                                                                     |
| ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------- |
| "Simpler for users to define collections of AI toolsets"                 | `sindri.yaml` is ~10 lines of `backend:name: version`. Collections are meta-components you depend on. |
| "Break apart coarse-grained extensions"                                  | Atomic components with `dependsOn` replace bundle extensions. `ai-toolkit` → 5 components.            |
| "User decides what package managers to employ"                           | Backend is the map key; the user picks explicitly per tool.                                           |
| "Our job is to manage registries per provider with a subset of software" | Per-backend registries (`registry-core`, `registry-ai`, …), curated and CI-enforced.                  |
| "We pin versions"                                                        | Registry entries are exact-pinned; resolver emits a lockfile with digests.                            |
| "Fine-grained software versions as a BOM to install post provisioning"   | Exactly `sindri.yaml` + `sindri.lock`.                                                                |
| "Breaking change, no backward compat"                                    | Clean slate; no migration path in scope.                                                              |

## 10. Biggest risks

1. **Registry publishing ergonomics.** If publishing a component to an OCI registry is harder than `git add extensions/foo && git push`, extension authors will grumble. Mitigation: `sindri registry publish` tooling that bundles the OCI workflow.
2. **OCI hard dependency.** Air-gapped users need an offline mirror. Already solvable (OCI mirrors, `skopeo copy`), but must be explicit in the docs from day one.
3. **Capability carryover.** Collision handling and project-init are tightly tied to today's extension lifecycle. The component-as-OCI-artifact change must not break their contracts — see `05-open-questions.md#3`.
4. **`Hybrid`-removal breakage.** Several v3 extensions (docker, guacamole) rely on the hybrid pattern. They become two-component compositions; the `script` component needs a way to depend on "the apt install completed" reliably. Likely fine with `dependsOn`, but needs validation during a prototype.
