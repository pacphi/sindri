# ADR-014: Signed Registries with cosign from Day One

**Status:** Accepted
**Date:** 2026-04-24
**Deciders:** sindri-dev team

## Context

v3 registry data (the git file `registry.yaml`) is unsigned. There is no way for a user
to verify that the registry they received is the one Sindri published, or that it has not
been tampered with in transit or at rest.

v4 distributes registries as OCI artifacts (ADR-003). OCI provides content-addressability
(manifest digests); cosign/sigstore provides signing. Both are necessary for a credible
security story.

## Decision

**All Sindri-published registries are signed with cosign from the first v4.0 release.**
Third-party registries cannot be added without an explicit `sindri registry trust` step
(unless added with `--no-verify`, which is logged to the StatusLedger).

### Signing pipeline (registry-publish CI)

```bash
# Step 4 of publish workflow (10-registry-lifecycle.md §6 step 4):
oras push ghcr.io/sindri-dev/registry-core:2026.05 ...   # push artifact
cosign sign ghcr.io/sindri-dev/registry-core:2026.05     # sign manifest digest
# SLSA provenance attestation generated by GitHub-generated build provenance
```

### Consumer verification

```bash
sindri registry trust acme --signer cosign:key=<path-or-k8s-ref>
```

`sindri registry refresh` verifies the cosign signature before writing to cache.
Signature failure aborts with the stale index left in place (fail closed, not open).

`sindri resolve` verifies component-blob digests against the index's declared digests.
Any mismatch is a hard error (not a warning).

`sindri apply` verifies downloaded binary artifacts against checksums in `component.yaml`.

Three independent integrity checks, each at a distinct layer boundary.

### Trust model

| Source                                                   | Trust level                                                                        |
| -------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| `sindri/core` (published by sindri-dev)                  | Always trusted (hardcoded public key)                                              |
| User-added registries                                    | Must explicitly `sindri registry trust <name>` with a cosign key                   |
| Private registries via `sindri registry add --no-verify` | Added without signature; logged as unverified; surfaced as warning at resolve time |

Open question Q7 resolved: signed registries required from day 1.

### Component-blob integrity

The registry `index.yaml` includes a `sha256:…` digest per component version. `sindri
resolve` verifies each `component.yaml` blob against its declared digest before using it.
Binary asset checksums are verified at download time during `sindri apply`.

## Consequences

**Positive**

- Supply-chain integrity for the registry layer — critical given that registries describe
  what to install on user machines.
- Enterprise users can pin the `sindri/core` public key and enforce it via policy.

**Negative / Risks**

- Cosign toolchain must be available or embedded. Mitigation: embed cosign verification
  as a Rust library (`sigstore-rs`) rather than shelling out.
- Key rotation requires a documented process. Mitigation: publish key rotation policy
  in `sindri.dev/security`.
