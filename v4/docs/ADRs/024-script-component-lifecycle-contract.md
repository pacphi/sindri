# ADR-024: Script Component Lifecycle Contract

**Status:** Accepted  
**Date:** 2026-04-24  
**Deciders:** sindri-dev team

## Context

Script-backend components are tools that cannot be installed by a native typed backend
(`mise`, `npm`, `binary`, etc.) and instead rely on shell scripts to manage their
lifecycle. Examples: `sdkman`, `docker`, `gcloud`, `playwright`, `supabase-cli`.

Unlike `mise:nodejs` (where `mise install node@24` is a single atomic operation), script
installs are heterogeneous — each tool has its own installer, versioning scheme, and
upgrade path. This raises several questions:

1. How does the CLI tell a script what version to install?
2. How should a script behave if the tool is already present (different version, same
   version, or broken install)?
3. What is the upgrade contract — is there a separate upgrade mechanism, or does
   install cover it?

## Decision

### 1. `SINDRI_COMPONENT_VERSION` — the version handoff

The sindri CLI injects `SINDRI_COMPONENT_VERSION` as an environment variable before
invoking any lifecycle script. Its value is the exact `metadata.version` from the
component.yaml that is being acted on.

Every script receives this contract:

```bash
WANT="${SINDRI_COMPONENT_VERSION:-<component-yaml-default>}"
```

The default (`<component-yaml-default>`) is baked into the script at generation time
from the registry component.yaml so that scripts remain runnable manually (outside of
`sindri`) without requiring the env var.

### 2. Version-aware idempotency — mandatory for all install scripts

Every `install.sh` MUST implement version-aware idempotency using `at_version`:

```bash
at_version() {
    local bin="$1" have
    have=$(version_of "$bin")
    if [[ -n "$WANT" && ( "$have" == "$WANT"* || "$have" == *"$WANT"* ) ]]; then
        info "$bin $WANT already installed. Nothing to do."
        return 0   # skip — already at target
    fi
    [[ -n "$have" ]] && info "$bin: $have → $WANT" || info "Installing $bin $WANT…"
    return 1       # proceed — install or upgrade
}
```

A script that exits 0 on presence alone (`if has tool; then … exit 0; fi`) is non-
conformant — it breaks `sindri upgrade` when a new registry version bumps the
component to a higher version.

### 3. `upgrade.sh` delegates to `install.sh`

**The install script IS the upgrade path** for script-backend components.

When `sindri upgrade script:docker` is run:

1. The CLI resolves the new component version in the registry.
2. Pulls the new OCI component package (which contains updated `install.sh`).
3. Sets `SINDRI_COMPONENT_VERSION=<new-version>`.
4. Runs `upgrade.sh`, which for most components does `exec install.sh`.

The new `install.sh` carries the logic to reach the new version. `upgrade.sh` is a
thin dispatch wrapper — not a separate upgrade strategy.

**Exception**: tools with an efficient native self-update mechanism may implement
`upgrade.sh` directly rather than re-downloading:

| Component  | upgrade.sh strategy                                                         |
| ---------- | --------------------------------------------------------------------------- |
| `sdkman`   | `sdk selfupdate force` — SDKMAN has a first-class self-update               |
| `gcloud`   | `gcloud components update --quiet` — Google manages its component manifest  |
| All others | `exec bash install.sh` — re-run install with new `SINDRI_COMPONENT_VERSION` |

### 4. Lifecycle script set

Every script-backend component ships exactly four scripts:

| Script         | Invoked by                   | Purpose                                      |
| -------------- | ---------------------------- | -------------------------------------------- |
| `install.sh`   | `sindri apply`, `sindri add` | Install or upgrade to `$WANT`                |
| `upgrade.sh`   | `sindri upgrade`             | Delegate to install.sh or native self-update |
| `uninstall.sh` | `sindri remove`              | Remove binary and optionally data (prompts)  |
| `validate.sh`  | `sindri validate`, CI        | Assert tool is present and at `$WANT`        |

### 5. Script generation

All scripts are generated by `tools/generate_scripts.py`. The shared header is
authoritative for the version contract helpers (`version_of`, `at_version`, `WANT`).
Per-component install logic is defined in the generator's `INSTALLS` dict.

When a component version bumps (e.g., `gcloud` 565.0.0 → 566.0.0):

- Update `metadata.version` in `decompose_extensions.py`
- Re-run `python3 tools/decompose_extensions.py` (regenerates component.yaml)
- Re-run `python3 tools/generate_scripts.py` (regenerates scripts with new version
  baked into `WANT` default)
- Publish the new component OCI package

The baked-in default in `WANT` is a safety net for manual script execution, not the
authoritative version source at runtime.

## Consequences

**Positive**

- `sindri upgrade script:X` works correctly because install.sh is the upgrade path
  and `at_version` skips no-ops.
- Script authors have a single place to express install logic; upgrade is free.
- Version-aware idempotency prevents redundant downloads when running `sindri apply`
  on an already-provisioned machine.
- `SINDRI_COMPONENT_VERSION` provides a clear, auditable handoff between the CLI and
  the script layer.

**Negative / Trade-offs**

- Script components cannot pin to sub-patch versions unless the install script
  explicitly downloads a specific release URL — the tool's bundled version manager
  may install the latest micro-patch even when given a minor version.
- Tools whose own upgrader fetches "latest" rather than `$WANT` (e.g., `goose update`)
  may not land on the exact version the registry declares. These are treated as
  best-effort; typed backends (mise, sdkman) should be preferred when version pinning
  is critical.

## Alternatives rejected

- **Separate upgrade.sh for every tool**: Adds complexity without benefit for most
  tools. The registry package publish cycle IS the upgrade mechanism.
- **Single `lifecycle.sh` with verb argument**: Would require a different CLI
  invocation convention and doesn't compose as cleanly with `exec` delegation.
- **Skip version check, always reinstall**: Slow for large tools (ollama ~800 MB) and
  makes `sindri apply` unusable on already-provisioned machines.
