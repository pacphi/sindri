# ADR-029: Lockfile Commit Policy

**Status:** Accepted
**Date:** 2026-04-30
**Deciders:** sindri-dev team

## Context

`sindri resolve` writes one lockfile per target (per ADR-018):
`sindri.lock` for the implicit `local` target and `sindri.<target>.lock`
for any other target. These files pin every component, source, and
auth-binding to byte-precise content addresses.

The original `sindri init` (pre-v4.0) appended `sindri.*.lock` to the
generated `.gitignore`, treating lockfiles as ephemeral build output.
The April 2026 docs/impl audit (F-CLI-10) flagged the inconsistency:
the public docs implied "appends `.sindri/` to `.gitignore`" while the
implementation also added the lockfile glob.

We considered two consistent positions:

1. **Don't commit lockfiles.** Treat `sindri.<target>.lock` like `.cache/`
   build output. CI regenerates them on each run. Diff noise stays out
   of code review.
2. **Commit lockfiles.** Mirror Cargo's behavior for binary crates:
   the lockfile is part of the project's reproducible-build contract,
   not output. CI consumes the committed lockfile rather than
   regenerating it.

## Decision

**Per-target lockfiles are committed to version control.** Mirrors
Cargo's `Cargo.lock` semantics for binary projects (`cargo new --bin`):
the lockfile is *the* statement of the resolved dependency graph that
shipped to a given commit, and reproducibility across developers and CI
depends on that graph being identical.

Concretely:

- `sindri init` only appends `.sindri/` to `.gitignore`. It does NOT
  add `sindri.*.lock`.
- `.sindri/` (the on-disk cache and trust state) remains ignored.
- Migration guides recommend that operators upgrading from older `init`
  output remove the `sindri.*.lock` line from `.gitignore` manually.
  Sindri does not auto-rewrite the file — touching a user's
  `.gitignore` outside of `init` is out of scope.

## Consequences

**Positive**

- A `git checkout <ref> && sindri apply` produces byte-identical
  installed state to whoever produced `<ref>`.
- CI pipelines can be `--strict-oci` and skip resolve (`sindri apply`
  alone) for fast, hermetic runs.
- The lockfile becomes a code-review artifact: dependency changes
  surface as diffs, not as silent re-resolutions.

**Negative / Risks**

- Lockfile diffs are large (every dependency, every checksum). Mitigation:
  `sindri resolve --json` and `git diff` work on lockfiles; reviewers
  can rely on the resolver's deterministic key ordering.
- Merge conflicts in lockfiles cannot be hand-resolved. The recovery
  procedure is: take either side, then re-run `sindri resolve` and
  commit. (Same as Cargo.)
- Existing v4 projects whose `.gitignore` was generated by an older
  `init` still list `sindri.*.lock`. The behavior change is documented
  in `MIGRATION_FROM_V3.md` and the v4.0 CHANGELOG; users must remove
  the line themselves.

## Alternatives considered

- **Branch-conditional behavior** (commit only on `main`, ignore on
  feature branches). Rejected — adds tooling complexity and weakens the
  reproducibility guarantee that motivated the decision.
- **Commit only the `local` lockfile, ignore non-local ones.** Rejected
  — non-local target lockfiles (e.g. `sindri.docker.lock`) are exactly
  the ones where reproducibility across CI runners matters most.
